## Принятые замечания ревьюеров (accepted, не блокеры)

Зарегистрированы по итогам реализации и review. Каждое принято владельцем/командой как приемлемый осознанный риск; действие — опциональное/будущее улучшение. TD-005..009 — minor, TD-010 — major (принят владельцем как осознанный риск).

| ID | Severity | Описание | Предлагаемое действие |
| --- | --- | --- | --- |
| `TD-005` | minor | **Redis durability.** Redis запущен без AOF (`--appendonly no`, только RDB `save 60 1`). При аварийном рестарте возможна потеря до ~60 с данных: активные сессии, pending-токены Mini App SSO, счётчики rate-limit/lockout. Принято: данные эфемерны, потеря = повторный логин / сброс лимитов. | При необходимости durability — включить `--appendonly yes` (AOF) в конфиге `redis`-сервиса ([07-deployment.md](./07-deployment.md)). |
| `TD-006` | minor | **Тег образа `:latest`.** В `docker-compose.yml` локально собираемые образы `migrate`/`app` помечены статичным `:latest` — нет трассируемости для отката к конкретной сборке. Принято: single-host деплой, откат через пересборку. | Опционально параметризовать тег: `${APP_IMAGE_TAG:-latest}`, привязывать к git sha/версии. |
| `TD-007` | minor | **Logout-CSRF без JS.** Форма выхода в `base.html` получает `csrf_token` только через `csrf.js`; при полностью отключённом JavaScript logout не пройдёт CSRF-проверку. Принято: UI и так требует JS (Mini App, admin_users.js). | Опционально прокидывать `csrf_token` в контекст SSR-страниц и рендерить hidden-полем формы logout. |
| `TD-008` | minor | **CSS-класс бейджа привязки.** В `admin_users.js` бейдж «привязан» переиспользует CSS-класс роли `--group_leader` вместо собственного модификатора. Косметика, семантически неверно. | Ввести отдельный модификатор `--linked` для бейджа Telegram-привязки. |
| `TD-009` | minor | **Остаточное покрытие тестами.** Непокрытые зоны при coverage 84%: Protocol-интерфейсы (`app/domain/repositories.py`), тонкие сетевые обёртки (`redis_client`, `telegram_api`), `main.py` lifespan. Принято: не бизнес-логика, покрытие не даёт ценности. | Действий не требуется; при рефакторинге сетевого слоя — добавить интеграционные тесты по необходимости. |
| `TD-010` | **major** | **Первичная установка пароля не защищена out-of-band секретом.** Актор, знающий существующий логин в состоянии `password_reset_required=true` (окно между созданием пользователя админом и первым легитимным входом; а также после admin reset), может занять неактивированный аккаунт, задав пароль на `/set-password`. Принято владельцем: флоу остаётся по ТЗ (админ создаёт только логин → пользователь входит под логином и сам задаёт пароль; без invite-токена и без обязательного Telegram-proof). **Амендмент [ADR-0002](./adr/ADR-0002-two-step-login.md) (2026-07-02):** шаг-1 `POST /login` теперь ветвится по состоянию и направляет неактивированный аккаунт сразу на `/set-password` (ТЗ-флоу «придумай пароль»), жертвуя жёсткой анти-энумерацией шага-1 — путь claim короче на один шаг, класс риска тот же (см. [08-security.md](./08-security.md) §6/§6a). Компенсирующие меры: секретность логина, rate-limit на `/login` и `/set-password` ([08-security.md](./08-security.md) §4), короткое окно активации, запрет reset для super_admin (`cannot_reset_super_admin`). | Возможное будущее ужесточение (не в этой итерации): требовать proof владения через Telegram Mini App на `/set-password`, либо одноразовый invite-токен, выдаваемый админом при создании пользователя. |
