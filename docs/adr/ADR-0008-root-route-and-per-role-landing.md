# ADR-0008. Корневой маршрут `/` и per-role landing-страницы

- Статус: accepted
- Дата: 2026-07-02
- Связано с: [ADR-0002](./ADR-0002-two-step-login.md) (двухэтапный логин), [ADR-0003](./ADR-0003-roles-and-teams.md) (роли), [ADR-0004](./ADR-0004-telegram-mini-app-sso.md) (Mini App SSO), [ADR-0005](./ADR-0005-sms-addressing-via-team.md) (номера через team).

## Context

На проде обнаружен функциональный баг + пробел:

1. **Битый пост-логин-редирект.** `POST /login/password`, `POST /set-password` и Mini App SSO (`POST /api/telegram/auth`, ветка «привязка существует») выполняют `303/redirect → /`. Но маршрут `GET /` **не зарегистрирован** — приложение отдаёт `404 {"error":"not_found"}`. После успешной аутентификации пользователь видит 404.
2. **Нет landing для участников.** SSR-страницы существуют только для `super_admin` (`/admin`, `/admin/teams`). У ролей `group_leader` / `group_member` нет ни `/`, ни собственной страницы. При этом по ТЗ ([01-overview.md](../01-overview.md) §«Основные сценарии» п.7, [ADR-0005](./ADR-0005-sms-addressing-via-team.md)) любой участник команды должен добавлять/удалять номера своей команды. API для этого есть (`/api/numbers` GET/POST/DELETE, guard `require_authenticated`), но UI-страницы, с которой участник его вызывает, нет — ранее numbers-UI был помечен вне scope, из-за чего участник после логина оказывался без достижимой страницы.

Итог: **не у каждой аутентифицированной роли есть достижимая стартовая страница**, а цель пост-логин-редиректа (`/`) физически отсутствует.

## Decision

### 1. Корневой маршрут `GET /` — диспетчер по роли

Регистрируется `GET /` (SSR-роутер), который **не рендерит контент**, а редиректит:

| Состояние | Ответ |
| --- | --- |
| Нет валидной `sms_session` | `302 → /login` |
| `role == super_admin` | `302 → /admin` |
| `role IN (group_leader, group_member)` | `302 → /app` |

`/` становится **единственной точкой диспетчеризации** landing по роли. Благодаря этому существующие `303 → /` в auth-флоу и `redirect:"/"` в Mini App SSO **остаются корректными** — они ведут на реальный маршрут, который сам доводит пользователя до страницы, достижимой для его роли. Отдельные per-role редиректы в auth-коде не размазываются: решение о landing принимается в одном месте (`GET /`).

### 2. Landing-страница участника — `GET /app` (SSR)

Выбрано имя **`/app`** (не `/numbers`), чтобы не путать SSR-страницу с API-ресурсом `/api/numbers` и оставить место под будущее расширение UI участника/лидера (управление участниками у лидера — потенциально позже).

- **Guard:** `require_authenticated`.
- **Роли:** `group_leader`, `group_member`. Для `super_admin` (нет команды, `team_id IS NULL`) `/app` не имеет смысла → `302 → /admin` (его landing). Так у **каждой** роли есть достижимая страница и нет тупиков.
- **Содержимое (SSR-контекст, scope = `current_user.team_id`):**
  - список номеров своей команды (тот же набор, что отдаёт `GET /api/numbers` для участника — по `VisibilityScope.team_id`);
  - форма добавления номера (`POST /api/numbers`) и удаление номера (`DELETE /api/numbers/{id}`) — по правам «любой участник управляет номерами своей команды» ([ADR-0005](./ADR-0005-sms-addressing-via-team.md));
  - статус собственной Telegram-привязки (`telegram_links` текущего user, `dead_at IS NULL` = привязан);
  - ссылка/форма logout.
- **Telegram-статус — без нового API.** Отдельного публичного endpoint для «мой статус привязки» не вводим: страница SSR, поэтому статус привязки текущего пользователя и стартовый список номеров **инжектируются в контекст сервером** (сервер уже знает `current_user` и его scope). Мутации (add/delete) идут через существующие `/api/numbers` (CSRF double-submit). Это минимизирует новую поверхность API и повторяет паттерн `/admin` + `admin_users.js`.
- **CSRF:** страница за сессией; `csrf_token` доступен на странице (hidden-поле/`csrf.js`), мутации `/api/numbers` требуют CSRF (см. [08-security.md](../08-security.md) §3).

### 3. Лидер на этой итерации = участник

Согласно [ADR-0003](./ADR-0003-roles-and-teams.md), на текущей итерации `group_leader` не имеет доп. привилегий над участниками (управление людьми — у `super_admin`). Поэтому `/app` для лидера и участника **идентична**. Управление участниками команды лидером — вне scope этой итерации (возможное будущее расширение, отражено в Consequences).

### 4. Инвариант достижимости (нормативно)

Вводится явное требование, проверяемое QA: **у каждой аутентифицированной роли есть landing, отдающий рендер `200` (после диспетчеризации `/`), и цель любого пост-логин / пост-set-password / SSO-редиректа существует и доступна роли.**

- `super_admin` → `/` → `/admin` → `200`.
- `group_leader` / `group_member` → `/` → `/app` → `200`.
- без сессии → `/` → `/login` → `200`.

## Rationale

- **Один диспетчер** (`GET /`) проще и надёжнее, чем правка каждого места редиректа: не нужно менять контракты `POST /login/password`, `POST /set-password`, `POST /api/telegram/auth` — они уже указывают на `/`.
- **`/app` вместо `/numbers`** — отделяет страницу от API `/api/numbers`, снижает риск коллизий и путаницы, оставляет пространство для роста UI участника.
- **Без нового API для статуса привязки** — SSR уже имеет доступ к данным; добавлять endpoint = лишняя поверхность и лишний CSRF/guard-код при нулевой выгоде.
- Решение не противоречит действующим ADR: роли и права (ADR-0003), права на номера (ADR-0005), поведение logout/SSO (ADR-0004) сохранены.

## Consequences

**Плюсы:**
- После логина ни одна роль не упирается в 404; у всех есть рабочая стартовая страница.
- Участник получает UI для управления номерами и видит статус своей Telegram-привязки.
- Точка принятия решения о landing одна — легко менять/тестировать.

**Минусы / принятые ограничения:**
- Добавляется SSR-страница `/app` + шаблон + клиентский JS (реализуют frontend/backend), покрытие тестами (qa).
- Управление участниками лидером не реализуется в этой итерации (см. §3) — остаётся возможным будущим улучшением; отдельного TD не заводим (это осознанный scope, а не долг), при запросе — новый ADR.

## Alternatives

1. **Править каждый редирект на конкретную страницу роли** вместо `/`-диспетчера. Отклонено: дублирует логику выбора landing в нескольких хендлерах, легче рассинхронизировать.
2. **Назвать страницу `/numbers`.** Отклонено: коллизия по смыслу с `/api/numbers`, хуже расширяемость.
3. **Ввести API «мой профиль/статус привязки» и рендерить страницу целиком на клиенте.** Отклонено: лишняя API-поверхность и CSRF/guard-код; SSR-инжект проще.
4. **Рендерить контент прямо на `GET /` (без `/app`).** Отклонено: `/` совмещал бы диспетчеризацию для admin и рендер для участника — менее чисто; отдельный `/app` явнее и тестируемее.
