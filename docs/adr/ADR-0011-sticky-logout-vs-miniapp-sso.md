# ADR-0011 — «Залипающий» logout против авто-SSO Mini App (cookie `sms_logged_out`)

| | |
| --- | --- |
| Статус | accepted |
| Дата | 2026-07-02 |
| Связано | ADR-0002 (двухэтапный логин), ADR-0004 (**амендит** §5 logout/SSO), ADR-0008 (диспетчер `/`); [05-api-contracts.md](../05-api-contracts.md) §2,§3,§7, [08-security.md](../08-security.md) §2,§3 |

## Context

На проде обнаружен функциональный баг: **logout не работает в Telegram Mini App**.

1. `POST /logout` корректно ревокает `sms_session` и чистит cookies, отвечает `302 → /login` ([05](../05-api-contracts.md) §2).
2. Но каждая SSR-страница наследует `base.html`, который подключает `tg.js`. При загрузке любой страницы (включая `/login`) `tg.js` **безусловно** POSTит `init_data` в `POST /api/telegram/auth`.
3. По [ADR-0004](./ADR-0004-telegram-mini-app-sso.md) §5 logout **не трогает** `telegram_links` (push переживает logout). Поэтому привязка жива, и ветка «нет сессии, привязка существует» endpoint'а `/api/telegram/auth` **сразу создаёт новую сессию** → пользователь мгновенно перелогинивается. Осознанный выход невозможен.

Требуется механизм, который отличает «пользователь только что осознанно вышел» от «пользователь открыл Mini App и хочет persistent-SSO», **не нарушая** ADR-0004 (привязка и push переживают logout).

## Decision

Вводится маркер «только что вышел» — cookie **`sms_logged_out`**, подавляющий авто-SSO до следующего **явного** входа.

### 1. Cookie `sms_logged_out`

- **Назначение:** флаг «сессия завершена осознанно, авто-SSO подавлено».
- **Атрибуты:** значение `1`; **НЕ HttpOnly** (читается `tg.js`, чтобы решить, POSTить ли initData); `Secure` при `COOKIE_SECURE`; `SameSite=Lax`; `Path=/`; TTL `LOGOUT_STICKY_TTL_SECONDS` (safety-net, см. §4).
- Маркер **не является** секретом и не влияет на push: `telegram_links` остаются нетронутыми (ADR-0004 §5 в силе). Он лишь подавляет автоматическое создание сессии.

### 2. `POST /logout` — ставит маркер

В дополнение к текущей логике (revoke `sms_session`, clear `sms_session`/`sms_csrf`, `telegram_links` не трогаются, audit `admin_logout` для super_admin) logout **устанавливает** `Set-Cookie sms_logged_out=1`. Ответ прежний — `302 → /login`.

### 3. Двойная защита (client + server)

**(a) `tg.js` (первичная защита).** Перед авто-POST в `/api/telegram/auth` скрипт проверяет наличие cookie `sms_logged_out`:
- **есть** → авто-SSO **НЕ выполняется** (initData не отправляется). Пользователь остаётся на `/login`, где показывается кнопка «Войти» (§«UX»).
- **нет** → авто-SSO как раньше (ADR-0004).

**(b) `POST /api/telegram/auth` (fallback).** Если запрос всё же пришёл (race / принудительный вызов) и в нём есть cookie `sms_logged_out` **при отсутствии** активной `sms_session`:
- ветки «привязка существует» и «привязки нет» **НЕ создают сессию и НЕ выдают pending** → `200 {"linked": false, "logged_out": true}` **без** `Set-Cookie sms_session`/`sms_tg_pending`. Пользователь гарантированно остаётся разлогиненным.
- ветка «есть активная `sms_session`» — маркер игнорируется (сессия новее выхода): выполняется `self_heal_link` как обычно, а stale-`sms_logged_out` **очищается** (`Set-Cookie sms_logged_out=; Max-Age=0`).

HMAC-валидация initData (§5 [08](../08-security.md)) выполняется в любом случае до ветвления; маркер проверяется после HMAC.

### 4. Жизненный цикл маркера

Маркер «залипает» **до следующего явного входа**; TTL — только safety-net.

Маркер сбрасывается (cookie удаляется) при любом из:
- **явный вход** — пользователь нажал «Войти» на `/login` (клиент удаляет cookie перед инициацией входа, §«UX»);
- **установление новой сессии** — успешный `POST /login/password`, `POST /set-password` или SSO после явного входа: вместе с `Set-Cookie sms_session`/`sms_csrf` сервер выдаёт `Set-Cookie sms_logged_out=; Max-Age=0`;
- **обнаружена активная сессия** в `/api/telegram/auth` (§3b) — stale-маркер чистится.

TTL `LOGOUT_STICKY_TTL_SECONDS` — **длинный** (рекоменд. дефолт 30 дней), т.к. первичный сброс — явный вход. Короткий TTL недопустим: Mini App переоткрывается часто, и при TTL в несколько минут пользователь, вернувшийся позже, был бы авто-залогинен — это выглядело бы как «logout не сработал». TTL нужен лишь чтобы cookie не жила вечно на устройстве, с которого больше не входят.

### 5. UX явного повторного входа

На `/login`, когда присутствует `sms_logged_out`:
- страница показывает сообщение «Вы вышли из системы» и **кнопку «Войти»** (вместо тихого авто-SSO);
- по клику клиент (`tg.js`): удаляет cookie `sms_logged_out` → затем инициирует вход:
  - **в Mini App** — POSTит `init_data` в `/api/telegram/auth` (маркер уже удалён → нормальное ветвление ADR-0004: живая привязка → сессия и `redirect:"/"`; нет привязки → `sms_tg_pending` + двухэтапный логин);
  - **в обычном браузере** (initData нет) — просто переход к форме шага-1 логина (`/login` без маркера).

Так выход — «залипающий» (повторный авто-SSO не происходит), но повторный вход — в один осознанный клик.

## Rationale

- **Двойная защита** (client skip + server refuse) устойчива к гонкам: даже если `tg.js` не успел прочитать cookie, сервер не создаст сессию при маркере.
- **Не нарушает ADR-0004:** `telegram_links` и push нетронуты; маркер влияет только на авто-создание веб-сессии. Push продолжает работать после logout (осознанный компромисс ADR-0004 сохранён).
- **`sms_logged_out` не HttpOnly** оправдано: как и `sms_csrf`, он должен читаться клиентом; он не секрет (лишь UX-флаг) и не даёт доступа — подмена/удаление злоумышленником максимум включит авто-SSO, что эквивалентно текущему (до фикса) поведению и всё равно требует валидного initData жертвы.
- **Длинный TTL + сброс по явному входу** даёт интуитивный UX: выход держится, вход — по кнопке.
- Диспетчер `/` (ADR-0008) не меняется: при маркере пользователь просто не получает сессию и остаётся на `/login`.

## Consequences

**Плюсы:**
- Logout реально работает в Mini App; пользователь остаётся на `/login` до осознанного входа.
- Push и привязки не затронуты (ADR-0004 в силе).
- Один явный клик для повторного входа.

**Минусы / издержки:**
- Новая cookie `sms_logged_out` и её очистка во всех точках установления сессии — нужно не забыть чистить (иначе пользователь «залипнет» на /login до истечения TTL). Покрывается тестами (qa).
- `tg.js` усложняется ветвлением (проверка маркера, кнопка «Войти»).
- Новый конфиг `LOGOUT_STICKY_TTL_SECONDS`.

## Alternatives

1. **Logout рвёт `telegram_links`.** Отклонено: прямо противоречит ADR-0004 §5 (push должен переживать logout); вернуло бы цикл «logout → нет push».
2. **Только client-side skip в `tg.js` (без серверной проверки).** Отклонено: гонки/обход JS снова залогинят пользователя; серверный fallback обязателен.
3. **Только серверный отказ (без флага в `tg.js`).** Отклонено: `tg.js` всё равно делал бы лишний POST на каждую загрузку; проверка на клиенте экономит запрос и даёт корректный UX кнопки «Войти».
4. **Redis-флаг «logged_out» по `telegram_user_id` вместо cookie.** Отклонено: `tg.js` не знает `telegram_user_id` до HMAC; потребовал бы round-trip и серверного состояния там, где хватает cookie на устройстве.
5. **Query-параметр `?logged_out=1` вместо cookie.** Отклонено: теряется при переоткрытии Mini App/навигации; cookie переживает перезагрузку страницы, что и требуется для «залипания».
