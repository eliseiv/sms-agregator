# ADR-0014. Просмотр SMS по номерам: ролевой доступ по текущей принадлежности номера + cursor-пагинация

- Статус: accepted
- Дата: 2026-07-03
- Связанные: [ADR-0005](./ADR-0005-sms-addressing-via-team.md) (адресация SMS через team), [ADR-0009](./ADR-0009-unassigned-numbers-admin-allocation.md) (unassigned-пул), [ADR-0012](./ADR-0012-multi-team-membership.md) (multi-team, `VisibilityScope.team_ids`), [ADR-0008](./ADR-0008-root-route-and-per-role-landing.md) (per-role landing).

## Context

До сих пор `inbound_sms` только принимались (webhook Twilio) и рассылались в Telegram; UI просмотра истории входящих SMS не было. Требуется страница, на которой:

- **super_admin** видит SMS всех номеров с опциональными фильтрами по конкретному номеру и по команде;
- **участник / лидер** (`group_member` / `group_leader`) видит SMS только тех номеров, что относятся к его командам.

Референс-паттерн (не копируется буквально) — mail-agregator, модуль messages: «роль → набор видимых контейнеров → фильтр по ним + cursor-пагинация». В нашей модели «контейнер» = номер телефона; связь SMS↔номер — по `inbound_sms.to_number = phone_numbers.phone_number`.

Три решения требуют фиксации, потому что затрагивают контракты, доступ и (впервые) HTTP-пагинацию:

1. **По какому признаку определять видимость участника** — по историческому снимку `inbound_sms.team_id` (проставляется в момент приёма, [04-data-model.md](../04-data-model.md) §`inbound_sms`) или по **текущей** принадлежности номера (`phone_numbers.team_id`).
2. **Только просмотр или мутации.** У `inbound_sms` нет полей прочитанности/ответа/удаления.
3. **Механика пагинации.** Это первое внедрение HTTP-пагинации в проекте — нужен зафиксированный контракт курсора.

## Decision

### 1. Единая страница `GET /messages` для обеих ролей

Одна SSR-страница обслуживает и super_admin, и участника; различие — только в наборе номеров ролевого селектора и в правилах видимости (см. §2). Пункт «Сообщения» добавляется в шапку (`base.html`) для **всех** аутентифицированных ролей. Отдельных `/admin/messages` и `/app/messages` не заводим — это дублировало бы визибилити-логику. JSON-данные отдаёт `GET /api/messages`; страница `GET /messages` рендерит первую страницу server-side (no-JS fallback обязателен) и прогрессивно обогащается JS.

### 2. Scope участника — по ТЕКУЩЕЙ принадлежности номера (`phone_numbers.team_id`), а не по снимку `inbound_sms.team_id`

Участник видит SMS номеров, которые **сейчас** принадлежат его командам:

```
inbound_sms.to_number IN (
    SELECT phone_number FROM phone_numbers WHERE team_id = ANY(:scope_team_ids)
)
```

где `:scope_team_ids` = `VisibilityScope.team_ids` (home ∪ доп. членства, [ADR-0012](./ADR-0012-multi-team-membership.md)).

**Почему не по `inbound_sms.team_id`:** снимок `team_id` фиксируется в `inbound_sms` в момент приёма ([03-architecture.md](../03-architecture.md) §«Поток приёма»). Если фильтровать по нему, участник **не увидит** недавние SMS по номеру, который назначили его команде уже **после** приёма этих SMS (типовой сценарий: номер добрали из unassigned-пула и распределили в команду — [ADR-0009](./ADR-0009-unassigned-numbers-admin-allocation.md)). Текущая принадлежность номера — актуальная модель владения, согласованная с адресацией доставки ([ADR-0005](./ADR-0005-sms-addressing-via-team.md)) и с `GET /api/numbers`, где номера тоже отдаются по текущему `phone_numbers.team_id`.

**Принятый trade-off:** участник может увидеть **исторические** SMS номера за период, когда номер принадлежал **другой** команде (или был unassigned) — если этот номер сейчас в его команде. Это принято сознательно: владение номером = владение его историей входящих. Обратная утечка (SMS номера, ушедшего из команды участника, остаётся видимым участнику) **исключена** — номер выпадает из `scope.team_ids`, и его SMS перестают быть видны. `inbound_sms.team_id` (снимок) для целей видимости **не используется**; остаётся как исторический аудит момента приёма.

**Super_admin** видит все `inbound_sms` без scope-ограничения, включая SMS, чей `to_number` уже не сопоставляется ни с одним `phone_numbers` (номер удалён). Фильтр `team_id` для super_admin трактуется так же — по **текущей** принадлежности номера (сопоставление `to_number ↔ phone_numbers.phone_number`, `phone_numbers.team_id = :team_id`), ради согласованности с участниковой семантикой.

### 3. Только просмотр (read-only)

Никаких mutate-операций над `inbound_sms`: нет отметки прочитанности, ответа, удаления, скрытия. `GET /api/messages` и `GET /messages` — единственные точки; схема `inbound_sms` не меняется (кроме индекса, см. ниже). Это устраняет CSRF-поверхность и упрощает контракт.

### 4. Cursor keyset-пагинация по `(received_at DESC, id DESC)` — первый пагинационный контракт проекта

- **Порядок:** `ORDER BY received_at DESC, id DESC`. `id` — детерминированный tie-breaker для строк с равным `received_at` (иначе keyset «залипает»/пропускает при равных таймстемпах).
- **Курсор — opaque:** `next_cursor` — непрозрачный для клиента `base64url`-токен (без padding), кодирующий **только позицию** — пару `(received_at, id)` последней отданной строки. Клиент не парсит и не конструирует его. Кодирование/декодирование — на сервере.
- **Курсор кодирует только позицию, не фильтры.** Фильтры (`to_number`, `team_id`, `limit`) клиент **пересылает** при каждом запросе рядом с `cursor`. Сервер не связывает курсор с набором фильтров → смена фильтров со «старым» курсором даёт валидную, но семантически смешанную страницу (ответственность клиента). Это осознанное упрощение против встраивания хэша фильтров и ошибки на рассинхрон.
- **Keyset-предикат** для страницы после `(r0, id0)`: `WHERE received_at < :r0 OR (received_at = :r0 AND id < :id0)` (эквивалент `(received_at, id) < (:r0, :id0)`).
- **`limit`:** целое, дефолт `50`, диапазон `[1, 100]`. Вне диапазона → `400 invalid_limit`.
- **Определение следующей страницы:** сервер читает `limit + 1` строк; если пришло `> limit`, лишняя отбрасывается, `next_cursor` строится из позиции **последней оставленной** (limit-й) строки. Если строк `≤ limit` — следующей страницы нет, `next_cursor = null`.
- **Битый курсор:** недекодируемый / структурно неверный токен → `400 invalid_cursor`. Пустой/отсутствующий `cursor` → первая страница.
- **Ответ:** `{"messages": [...], "next_cursor": "<opaque>" | null}`. Пагинация **forward-only** (keyset): «назад» не поддерживается, возврат к началу = запрос без `cursor`. SSR-страница рендерит ссылку «Дальше» (next), только когда `next_cursor != null`.

## Consequences

- Новый read-only контракт `GET /api/messages` (JSON) + SSR `GET /messages`; контракт cursor-пагинации переиспользуем как эталон для будущих листингов ([05-api-contracts.md](../05-api-contracts.md) §9).
- Видимость участника считается через сопоставление `to_number ↔ phone_numbers.phone_number` + `phone_numbers.team_id`. Требует индекс на `inbound_sms(to_number, received_at DESC, id DESC)` для эффективной фильтрации по номерам + keyset (добавляется этой доработкой, [04-data-model.md](../04-data-model.md) §`inbound_sms`). Глобальный путь super_admin (без фильтра по номеру) обслуживается существующим `ix_inbound_sms_received_at`.
- Запрос участником `to_number` **вне** своего scope не раскрывает существование SMS: результат — пустой список (`200`), а не `403`/`404` (анти-энумерация номеров чужих команд).
- `raw_payload` (полный payload webhook Twilio) в сериализации **не отдаётся** — только безопасное подмножество полей (§`serialize` в [05-api-contracts.md](../05-api-contracts.md) §9), чтобы не раскрывать служебные/потенциально чувствительные данные Twilio.
- Снимок `inbound_sms.team_id` теперь имеет **двойное** назначение: (а) исторический аудит команды на момент приёма; (б) для видимости **не** используется. Это зафиксировано, чтобы не возникло соблазна фильтровать по нему.

## Alternatives (rejected)

- **Видимость по снимку `inbound_sms.team_id`.** Отвергнуто: участник теряет недавние SMS номера, назначенного его команде после приёма (основной сценарий добора номеров из пула). Ломает интуицию «мой номер → моя история».
- **Отдельные страницы `/admin/messages` и `/app/messages`.** Отвергнуто: дублирование визибилити-логики и шаблонов; единая страница с ролевым селектором проще.
- **Offset/limit-пагинация (`?page=N`).** Отвергнуто: дрейф и дубли/пропуски при вставке новых SMS во время листания (входящие постоянно добавляются в голову), рост стоимости `OFFSET`. Keyset стабилен к вставкам в голову и дёшев.
- **Курсор со встроенным хэшем фильтров + `409` на рассинхрон.** Отвергнуто как избыточное для read-only просмотра; позиционного курсора и требования пересылать фильтры достаточно.
- **Мутации (прочитано/архив).** Отвергнуто: в `inbound_sms` нет соответствующих полей, а вводить их вне scope этой доработки; просмотр самодостаточен.
