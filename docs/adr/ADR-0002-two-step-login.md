# ADR-0002 — Двухэтапный логин (логин → пароль)

| | |
| --- | --- |
| Статус | accepted |
| Дата | 2026-07-01 |
| Связано | ADR-0004; [05-api-contracts.md](../05-api-contracts.md) §2, [08-security.md](../08-security.md) |

## Context

Админ-панель требует формы входа. Пользователи создаются админом с указанием **только логина** — пароль пользователь задаёт при первом входе (`password_hash IS NULL`, `password_reset_required=true`). Нужен flow, который:

- различает «первый вход / сброшенный пароль» (→ установка пароля) от обычного входа;
- не раскрывает существование логина (анти-энумерация);
- совместим с Telegram Mini App SSO ([ADR-0004](./ADR-0004-telegram-mini-app-sso.md)), где после ввода логина/пароля создаётся привязка из pending-токена;
- переносим 1:1 с референса mail-agregator (тот же двухэтапный flow).

## Decision

Ввести **двухэтапный логин**, разнесённый на отдельные endpoints (SSR):

1. **Шаг 1 — логин.** `GET/POST /login`. Пользователь вводит `username`. Backend нормализует (lower), выполняет `lookup_for_login`, ставит промежуточную cookie-сессию `sms_login` (короткий TTL) и **всегда** отвечает `303 → /login/password` — одинаково для существующего и несуществующего логина.
2. **Шаг 2 — пароль.** `GET/POST /login/password`. Пользователь вводит `password`. Backend `login()`: argon2 verify (с anti-timing dummy-hash для несуществующего/без пароля), учёт `failed_login_attempts`/`lockout_until`, `needs_rehash`. Ветвление:
   - `password_reset_required=true` или `password_hash IS NULL` → `303 → /set-password` (cookie `sms_setup`);
   - успех → создать сессию `sms_session`/`sms_csrf`, при наличии `sms_tg_pending` — `link_pending`; `303 → /`.
3. **Установка пароля.** `GET/POST /set-password` — задать пароль (политика — [08-security.md](../08-security.md)), `password_reset_required=false`, создать сессию; `link_pending` если есть pending.

Cookies: `sms_login` (шаг-1), `sms_setup` (установка пароля), `sms_session`/`sms_csrf` (после входа).

## Rationale

- **Анти-энумерация:** разнесение на два шага позволяет шагу-1 отвечать идентично при любом логине; факт существования не утекает. Ошибка появляется только на шаге-2 как общий `invalid_credentials`.
- **Первый вход / reset:** естественно вписывается как ветка после проверки логина/флага без отдельных экранов «активации».
- **Совместимость с Mini App SSO:** pending-токен потребляется в конце обоих flow (`/login/password` и `/set-password`), одинаково.
- **Перенос:** совпадает с проверенным flow mail-agregator (ADR-0016 там же), auth-код переносится 1:1.

## Consequences

**Плюсы:** защита от энумерации, чистое разделение «первый вход vs обычный», переносимость.

**Минусы:** на один HTTP-переход больше при входе; нужны промежуточные cookie-сессии (`sms_login`, `sms_setup`) с корректными TTL и очисткой. Rate-limit и lockout применяются на обоих шагах (см. [08-security.md](../08-security.md)).

## Alternatives

1. **Одностраничный логин (логин+пароль вместе).** Отклонено: сложнее обеспечить анти-энумерацию (нужно единое сообщение, но ветка «установить пароль» смешивается с «неверный пароль»); расходится с референсом.
2. **Basic-Auth / `X-Admin-Token` (текущее).** Отклонено: не поддерживает пользователей/команды/первый вход/Mini App SSO; удаляется.
3. **Magic-link / OTP.** Отклонено: избыточно для внутренней панели; нет почтового/SMS-канала для админ-логина; усложняет UX.

## Amendment (2026-07-02) — ветвление шага-1 по состоянию аккаунта; отмена анти-энумерации шага-1

> Тело Decision выше **иммутабельно**; данный раздел уточняет и частично пересматривает поведение шага-1. Связано: [05-api-contracts.md](../05-api-contracts.md) §2, [08-security.md](../08-security.md) §6/§6a, [100-known-tech-debt.md](../100-known-tech-debt.md) TD-010.

**Контекст правки.** Прежняя реализация (анти-энумерация, «MAJOR-2») делала шаг-1 `POST /login` **безусловным** `303 → /login/password` без проверки состояния аккаунта. В результате новый пользователь (`password_hash IS NULL` / `password_reset_required=true`) видел форму **ввода** пароля («введите пароль») и не мог войти, тогда как по ТЗ он должен **придумать** пароль. Это ломало основной сценарий первого входа.

**Решение владельца (приоритет ТЗ над анти-энумерацией шага-1).** Чинится ТЗ-флоу первого входа; полной анти-энумерацией на шаге-1 жертвуем — это согласуется с уже принятым остаточным риском [TD-010](../100-known-tech-debt.md) (актор, знающий логин в состоянии `password_reset_required`, и раньше мог занять аккаунт через `/set-password`; правка лишь укорачивает путь на один шаг, не создавая нового класса риска).

**Новое поведение шага-1 `POST /login`** (нормализует username → `lookup_for_login` → ветвление по состоянию):

| Состояние аккаунта по username | Ответ шага-1 |
| --- | --- |
| `password_hash IS NULL` **или** `password_reset_required=true` | создать setup-сессию + `Set-Cookie sms_setup` → `303 → /set-password` («придумайте пароль») |
| активный аккаунт с паролем | `303 → /login/password` |
| логин **не существует** | `303 → /login/password` (мягкая анти-энумерация) |

**Мягкая анти-энумерация (что сохраняется).** Различимо **только** состояние `password_reset_required` (→ `/set-password`) — это и есть принятый риск TD-010. Несуществующий логин отвечает **идентично** нормальному активному аккаунту (`303 → /login/password`), а шаг-2 для него даёт общий `invalid_credentials` через anti-timing dummy-hash. То есть факт **существования** активного аккаунта по-прежнему не раскрывается; раскрывается лишь факт «аккаунт в состоянии первичной активации», что уже принято владельцем.

**Шаг-2 `login()` — fallback сохраняется.** Ветка «`password_reset_required=true` / `password_hash IS NULL` → `/set-password` (`sms_setup`)» на шаге-2 **остаётся** как страховка: если состояние аккаунта изменилось между шагами или пользователь попал на `/login/password` иначе, шаг-2 всё равно перенаправит на установку пароля. Нормальный вход (verify argon2), lockout (`account_locked`/423), неверный пароль (`invalid_credentials`), rate-limit (`429`) — **не меняются**.

**CSRF.** `POST /login` остаётся CSRF-exempt (на шаге-1 сессии/`sms_csrf` ещё нет; защита — rate-limit). Setup-сессия `sms_setup`, выданная на шаге-1, несёт парный CSRF-cookie так же, как раньше при выдаче на шаге-2 — `POST /set-password` требует CSRF без изменений.
