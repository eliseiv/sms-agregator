# ADR-0012 — Членство пользователя в нескольких командах (multi-team, аддитивная M:N `user_teams`)

> **Отмена частного решения ADR-0003.** Настоящий ADR **отменяет только** [ADR-0003](./ADR-0003-roles-and-teams.md) §Decision 4 («Multi-team не вводится… `TD-003`») и §Alternatives 2 (отказ от M:N). Всё остальное в ADR-0003 (роли-константы `super_admin`/`group_leader`/`group_member`, инвариант роль↔команда, правило «первый=лидер», UNIQUE-лидер, жизненный цикл лидерства, циклический FK) остаётся **accepted** и в силе. Формально: **ADR-0012 supersedes ADR-0003 §4 (partial)** — в части запрета multi-team.

## Context

По [ADR-0003](./ADR-0003-roles-and-teams.md) пользователь принадлежит **ровно одной** команде: `users.team_id` (один FK) + табличный CHECK `users_role_team_invariant` (`group_member`/`group_leader` → `team_id NOT NULL`; `super_admin` → `team_id IS NULL`). Адресация SMS ([ADR-0005](./ADR-0005-sms-addressing-via-team.md) §2) и видимость номеров строятся на сравнении `users.team_id = :team_id`. ADR-0003 §4 явно объявлял multi-team out of scope, а §Alt.2 отвергал M:N `user_teams` как преждевременный (`TD-003`).

Продукт теперь **запросил** возможность для пользователя состоять в **нескольких командах одновременно** (по образцу mail-agregator ADR-0030 «Multi-group membership»):
- На `/admin` у пользователя — действие «**Добавить в другую команду**» (доп. членство, не меняя домашнюю) в дополнение к «**Переместить**» (сменить домашнюю команду).
- Участник нескольких команд должен **видеть/добавлять номера всех** своих команд в `/app` **и** получать в Telegram SMS **всех** своих команд (видимость консистентна с доставкой).

Сценарий, отвергнутый в ADR-0003 §Alt.2 как «не запрашивался», теперь подтверждён — решение пересматривается.

## Decision

Вводим аддитивную M:N-таблицу `user_teams` как **источник истины для адресации SMS и видимости номеров**, при этом `users.team_id` **сохраняется** в роли «домашней»/первичной команды. Дизайн минимально инвазивен и обратно совместим.

### 1. Аддитивная таблица `user_teams`; `users.team_id` остаётся «домашней» командой

`user_teams` (M:N `users` ↔ `teams`) становится **единственным источником истины** для:
- адресации SMS — `recipients_for_team(team_id)` (вместо `users.team_id = :team_id`; амендмент [ADR-0005](./ADR-0005-sms-addressing-via-team.md) §2);
- видимости/добавления номеров в `/app` и в `GET /api/numbers` для участника (номера **всех** своих команд, а не только домашней);
- подсчёта членов команды (`members_count`, список участников на `/admin`).

`users.team_id` (один FK) **сохраняется** как «домашняя»/первичная команда. Она по-прежнему нужна для:
- **инварианта лидера** — CHECK `users_role_team_invariant` + constraint-триггер лидерства ([ADR-0003](./ADR-0003-roles-and-teams.md) §6, [04-data-model.md](../04-data-model.md)) остаются на `users.team_id` (**не** на `user_teams`);
- **правила «первый=лидер»** и UNIQUE-лидера — на домашней команде;
- **глобальной роли** `users.role` — **пер-командные роли НЕ вводятся**, роль остаётся одна на пользователя;
- **дефолта команды при добавлении номера** — если участник не выбрал команду явно, номер попадает в домашнюю.

Структура таблицы, инварианты, DDL и backfill — [04-data-model.md](../04-data-model.md) (таблица `user_teams`, ревизия `20260702_003_user_teams`).

### 2. Видимость консистентна с доставкой

Участник N команд видит и добавляет номера **всех** своих команд (домашняя + добавленные) **и** получает по каждой из них SMS в Telegram. Никакой асимметрии «что вижу» ↔ «что доставляют»: обе модели читают `user_teams`. Пользователь в нескольких командах **автоматически** получает SMS каждой своей команды — без дополнительных подписок.

### 3. `phone_numbers.team_id` не меняется — номер принадлежит одной команде

Принадлежность номера команде остаётся 1:1 (`phone_numbers.team_id`, [ADR-0005](./ADR-0005-sms-addressing-via-team.md)/[ADR-0009](./ADR-0009-unassigned-numbers-admin-allocation.md)). Multi-membership означает: участник **видит номера ДРУГИХ своих команд**, а не то, что номер принадлежит нескольким командам. Дублирование `phone_numbers` под каждое членство отвергнуто.

### 4. Кого можно добавлять; статус лидера

- Добавлять в команды можно **всех, КРОМЕ `super_admin`** (он и так видит всё; членства бессмысленны и нарушили бы инвариант `super_admin → team_id IS NULL` и «нет строк в `user_teams`»). Попытка → `400 cannot_add_super_admin_to_team`.
- **Лидер** остаётся лидером **своей** (домашней) команды и может быть **дополнительно** участником других команд. Доп. членства лидера **не** делают его лидером тех команд (роль глобальна и одна).
- Обычные участники (`group_member`) — в **любом** числе команд.

### 5. Home-членство и синхронизация с `users.team_id`

- `users.team_id` (домашняя команда) **всегда зеркалируется** строкой в `user_teams`. Инвариант: для каждого не-`super_admin` с `team_id IS NOT NULL` существует ровно одна home-строка `user_teams(user_id, team_id=users.team_id)`.
- `create_user` (§4 контрактов): помимо проставления `users.team_id`, вставляет home-строку в `user_teams` в той же транзакции.
  - **Создание сразу в несколько команд (амендмент, 2026-07-06).** `POST /api/admin/users` принимает опциональный `extra_team_ids: list[int]` — доп. команды, в которые пользователь добавляется **при создании** (доп. членства `user_teams`, помимо домашней). В той же транзакции после home-членства + «первый=лидер» (только для home) вставляются доп. членства (дедуп, исключая домашнюю; несуществующая команда → `404 team_not_found`, откат). Доп. членства **не** делают лидером и не меняют роль (§4 настоящего ADR). Ревокация сессий не нужна (новый пользователь). Контракт — [05-api-contracts.md](../05-api-contracts.md) §4 (`POST /api/admin/users`, поле `extra_team_ids`).
- `PATCH /api/admin/users/{id}` (move — смена домашней команды): в одной транзакции **удаляет старую home-строку** (`team_id=старый users.team_id`) и **добавляет новую home-строку** (`team_id=новый`); доп. членства **не трогает**. Запрет переноса лидера непустой команды ([ADR-0003](./ADR-0003-roles-and-teams.md) §7) сохраняется.
- **Home-членство нельзя удалить** через membership-API: `DELETE /api/admin/users/{id}/teams/{team_id}` при `team_id == users.team_id` → `400 cannot_remove_home_membership`. Смена домашней команды — только через `PATCH` (move). Так CHECK `users_role_team_invariant` (у member/leader всегда есть home) не нарушается.

### 6. Ревокация сессий при изменении членства

Любая мутация членства — add / remove / move — **обязана** вызвать `SessionStore.revoke_all_for_user(target_user_id)` (как уже делается при `reset`/смене команды). Иначе `VisibilityScope.team_ids` в активной сессии останется stale, и участник продолжит видеть/не-видеть номера по устаревшему набору команд. После ревокации scope перечитывается из БД при следующем входе.

**Удаление команды (`DELETE /api/admin/teams/{id}`) — ревок доп.-участников.** `ON DELETE CASCADE` на `user_teams.team_id` при удалении команды снимает **все** строки членства в ней, включая доп.-членства пользователей, чья домашняя команда — другая. Для каждого такого пользователя (у кого была строка `user_teams` в удаляемой команде) `DELETE` команды **обязан** вызвать `revoke_all_for_user(user_id)` — иначе их `VisibilityScope.team_ids` продолжит содержать id удалённой команды до re-login. Практически: перед/в транзакции удаления команды собрать `list_user_ids_in_team(team_id)` (по `user_teams`) и ревокнуть сессии каждого. Домашние участники к моменту `DELETE` уже отсутствуют (disband-flow, [ADR-0003](./ADR-0003-roles-and-teams.md) §7); ревок покрывает оставшихся доп.-участников.

### 7. `VisibilityScope` несёт множество команд

`VisibilityScope` дополняется полем `team_ids: frozenset[int]` (домашняя + добавленные), наполняемым из `user_teams` при построении scope. Существующее одиночное поле-«домашняя» сохраняется для операций записи (дефолт команды при добавлении номера, инвариант лидера). `super_admin` → `team_ids` пуст (видит всё вне scope-фильтра). Детали построения scope и закрытия read-транзакции — [03-architecture.md](../03-architecture.md), контракт — [05-api-contracts.md](../05-api-contracts.md).

## Rationale

- **Минимально инвазивно и обратно совместимо.** Одна новая таблица (`user_teams`), `users.team_id` сохраняется, инварианты/триггеры лидера не трогаются. Backfill автоматически заполняет home-членства.
- **Единый источник истины.** Адресация SMS и видимость номеров читают одну таблицу `user_teams` — без рассинхронов «вижу ↔ получаю».
- **Роль остаётся глобальной и простой.** Без пер-командных ролей и матриц прав — прямой перенос кода ADR-0030.
- **Явное требование продукта** («участник в нескольких командах видит и получает всё») выражается точно через membership-JOIN.

## Consequences

**Плюсы:** обратная совместимость, единый источник истины видимости/доставки, роль остаётся одна на пользователя, прямой порт mail-agregator.

**Минусы / издержки:**
- **Усложнение SQL.** Предикат `users.team_id = :team_id` в `recipients_for_team` и в member-фильтрах заменяется membership-JOIN/`EXISTS` по `user_teams`. Счётчики/списки (`member_counts`, `list_by_team`, `list_user_ids_in_team`) переводятся на `user_teams`.
- **`VisibilityScope` получает множество `team_ids`** — построение scope дополнительно читает `user_teams`; scope-фильтр номеров переходит с одиночного `team_id` на `team_id = ANY(:team_ids)`.
- **Ревокация сессий при изменении членства** (§6) — обязательна, иначе stale scope.
- **Пользователь на `/admin` показывается в нескольких секциях** (в каждой своей команде; home помечен) — группировка на `/admin` строится из bulk-членств.
- **Дублирование доставки исключено на уровне идемпотентности:** участник, состоящий в командах владельца одного номера, получает SMS ровно один раз (UNIQUE `deliveries(inbound_sms_id, telegram_user_id)`); членство влияет только на **набор** команд-источников.

## Alternatives

Запрет multi-team из [ADR-0003](./ADR-0003-roles-and-teams.md) §4/§Alt.2 **пересмотрен** — сценарий подтверждён продуктом.

1. **Пер-командные роли (роль своя в каждой команде).** Отвергнуто. Продукт не запрашивал асимметрию прав по командам; видимость симметрична во всех командах участника. Потребовало бы вынести `role` в `user_teams` и переписать авторизацию — несоразмерно. Роль остаётся **глобальной** (`users.role`).
2. **Дублирование `phone_numbers` под каждое членство.** Отвергнуто. Нарушает 1NF, ломает UNIQUE `phone_number`, порождает неоднозначность «чей номер». Видимость через членство решает то же одним фильтром; `phone_numbers.team_id` остаётся 1:1.
3. **Массив `team_id BIGINT[]` в `users`.** Отвергнуто в пользу нормализованной `user_teams`. Массив не даёт FK-целостности (`ON DELETE CASCADE` при удалении команды), усложняет обратный поиск «члены команды» (GIN вместо B-tree), ломает реляционные JOIN. Нормализованная M:N с `UNIQUE(user_id, team_id)` + `INDEX(team_id)` — стандартное целостное решение.
4. **Удалить `users.team_id`, оставить только `user_teams`.** Отвергнуто. `users.team_id` нужен для инварианта лидера (CHECK+trigger [ADR-0003](./ADR-0003-roles-and-teams.md) §6), правила «первый=лидер», дефолта команды при добавлении номера. Аддитивный подход (home = `users.team_id` + `user_teams`) дешевле и безопаснее.
