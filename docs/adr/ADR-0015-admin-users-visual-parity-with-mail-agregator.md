# ADR-0015. Визуально-структурный паритет `/admin` (список пользователей) с референсом mail-agregator

- **Статус:** accepted
- **Дата:** 2026-07-04
- **Связано с:** [ADR-0003](./ADR-0003-roles-and-teams.md) (роли/команды), [ADR-0012](./ADR-0012-multi-team-membership.md) (multi-team), [ADR-0009](./ADR-0009-unassigned-numbers-admin-allocation.md)/[ADR-0013](./ADR-0013-on-demand-twilio-number-sync.md) (секция номеров). Контракт — [05-api-contracts.md](../05-api-contracts.md) §7 (`GET /admin`). Banding — [03-architecture.md](../03-architecture.md) §«Подсветка команд».

## Context

Требование заказчика с самого начала проекта: страница `/admin` (список пользователей) должна **точь-в-точь** повторять админку проекта-референса **mail-agregator** (`backend/app/templates/admin/users.html`, `backend/app/static/css/main.css`). В текущей реализации sms-агрегатора накопились структурные отклонения от референса, на которые заказчик указал прямо:

1. **Лишняя колонка «Telegram»** (`admin-users-table__tg`) — в референсе её нет (mail-agregator — email-сервис без Telegram).
2. **«Разделённый» дизайн команд** — sms рендерит **отдельную `<section>` с собственной `<table>`/`<thead>` на каждую команду** и **отдельную секцию «Администраторы»**; референс использует **одну таблицу** `admin-users-table`, где команды — это `<tbody>`-бакеты с чередующейся подсветкой, а super_admin входит в тот же список (бакет «без команды»).
3. **Дублирование пользователя** — при multi-team sms показывает пользователя **строкой в каждой команде** (row-per-membership); референс показывает пользователя **одной строкой**, а все его команды — **чипами** (`team-chip`) в колонке «Команда» (home-чип без «×», доп.-членства с «×»).
4. Прочие отклонения: наличие per-group заголовков/счётчиков («N участник(ов)»), иной набор/размещение контролов (инлайн-кнопки Сброс/Перевести/+команда/Удалить вместо «+»-меню управления командами в ячейке имени), отсутствие поиска и пагинации, иной состав тулбара.

Расхождение возникло из-за того, что контракт [05](../05-api-contracts.md) §7 фиксировал sms-специфичную модель группировки (`super_admins` + `team_sections` + row-per-membership), а не порт визуальной модели референса.

## Decision

**Визуально-структурный паритет `/admin` с mail-agregator принимается как обязательное приёмочное условие** (acceptance criterion) для frontend и reviewer. Целевая модель отображения списка пользователей — **порт `mail-agregator/backend/app/templates/admin/users.html`**:

1. **Одна таблица** `admin-users-table` в обёртке `user-groups-list` с **единым `<thead>`**. Колонки строго в порядке: **Имя · Роль · Команда · Создан · Последний вход · Действия** (6 колонок). Колонка «Telegram» **удаляется**.
2. **Команды — это `<tbody>`-бакеты** (`user-group user-group--band-a` / `--band-b`), между бакетами — спейсер-`<tbody class="user-group__spacer">`. Banding чередуется только по командным бакетам; бакет «без команды» (super_admin) — нейтральный `user-group--no-group` и рендерится **первым**. Per-group заголовков/счётчиков **нет** — принадлежность к команде передаётся цветом банда и чипами в колонке «Команда».
3. **Один пользователь = одна строка** в бакете его **домашней** команды (super_admin — в бакете «без команды»). Все команды пользователя — **чипами** в колонке «Команда» (`team-chips`/`team-chip`); домашний чип (`team-chip--home`) без контрола удаления, доп.-членства — с «×» (форма удаления с no-JS fallback `_method=DELETE`). Дублирование строк на членство **упраздняется**.
4. **Управление командами пользователя** — кнопка «+» (`admin-users-table__add-group`) в ячейке имени, открывающая меню-диалог «Переместить в другую команду» / «Добавить в другую команду» (для лидера «Переместить» задизейблено). Колонка «Действия» = **Сброс** + **Удалить** (super_admin → «системный»).
5. **Тулбар, поиск, пагинация** приводятся к референсу (детали и границы — §7).

**Сохраняются sms-специфичные элементы, которых нет в референсе** (легитимные дополнения, не нарушают паритет списка пользователей):
- Секция **«Нераспределённые номера»** ([ADR-0009](./ADR-0009-unassigned-numbers-admin-allocation.md)/[ADR-0013](./ADR-0013-on-demand-twilio-number-sync.md)) — ниже таблицы пользователей.
- **Semantics** создания/перевода/добавления пользователя остаются по sms-модели ролей ([05](../05-api-contracts.md) §4, ADR-0003 «первый=лидер», ADR-0012): диалог «Создать пользователя» требует `team_id` и **не** содержит radio-выбора роли (в отличие от референса) — копируется **визуальный паттерн** (диалоги, «+»-меню), **не** семантика создания.

**Telegram-привязка (`has_telegram_link`)** — sms-специфичные данные, которых нет в mail-agregator. **Финальное решение заказчика (Q-ADMIN-1, вариант «a», 2026-07-04): индикатор привязки на `/admin` не отображается вовсе** — ни колонки, ни бейджа, ни иконки. Поле `has_telegram_link` не входит в строку пользователя SSR-контекста `/admin` ([05](../05-api-contracts.md) §7). Данные привязки сохраняются в БД и доступны в других endpoint'ах — только на этой странице не показываются.

## Rationale

- Заказчик явно и с начала проекта требует пиксельно-структурного совпадения; отклонения — регресс относительно согласованного эталона.
- Единая таблица с `<tbody>`-бандингом + чипы = каноничная модель референса; она же корректно и без дублирования отображает multi-team (ADR-0012), который в mail-agregator уже реализован тем же способом.
- Паритет фиксируется как **acceptance criterion**, а не как «пожелание», чтобы reviewer имел объективный чек-лист и не пропускал «улучшения», расходящиеся с эталоном.

## Consequences

- **Контракт [05](../05-api-contracts.md) §7 `GET /admin` перепроектирован**: SSR-контекст переходит с `super_admins`+`team_sections`(row-per-membership) на `user_groups`(бакеты по домашней команде) + один пользователь со списком `memberships` (чипы). Литеральные имена — §7 (split-ownership SSR: шаблон авторит frontend, контекст — backend).
- **Backend**: `AdminService`-группировка перестраивается (линейный проход по предсортированному списку → бакеты по home-команде; `memberships` из `user_teams`); добавляются параметры `q`/`page`/`limit` к `GET /admin`.
- **Frontend**: `app/api/templates/admin/users.html` и соответствующие блоки `main.css` переписываются под порт референса (классы `user-groups-list`, `user-group--band-a/-b/--no-group`, `user-group__spacer`, `team-chips`/`team-chip`, `admin-users-table__add-group`); колонка `admin-users-table__tg` и per-group `admin__group-head`/секции удаляются.
- **Схема БД не меняется** (banding и чипы — презентация; данные — из существующих `users`/`user_teams`/`teams`).
- **QA**: приёмочный чек-лист — §7 «Паритет `/admin` (нормативно)».

## Alternatives

- **Оставить sms-модель (секции-на-команду + колонка Telegram)** — отклонено: прямо противоречит требованию заказчика о точь-в-точь паритете.
- **Сохранить колонку «Telegram»** — отклонено для паритета; вопрос индикатора вынесен заказчику (не удалять данные молча).

## Open question

- **Q-ADMIN-1 — resolved (2026-07-04):** заказчик выбрал вариант **(a) — убрать индикатор Telegram-привязки полностью**. На `/admin` `has_telegram_link` не отображается вовсе (ни колонки, ни бейджа, ни иконки) и не входит в SSR-контекст страницы. См. [99-open-questions.md](../99-open-questions.md) `Q-ADMIN-1` и [05](../05-api-contracts.md) §7.
