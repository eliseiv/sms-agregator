# ADR-0016. SSR-паритет страницы `/admin/teams` (список команд) с референсом mail-agregator

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Связано с:** [ADR-0015](./ADR-0015-admin-users-visual-parity-with-mail-agregator.md) (паритет `/admin` — тот же принцип «копируем визуальный паттерн, не семантику»), [ADR-0003](./ADR-0003-roles-and-teams.md) (роли/команды, «первый=лидер»), [ADR-0012](./ADR-0012-multi-team-membership.md) (multi-team, `members_count` по `user_teams`), [ADR-0009](./ADR-0009-unassigned-numbers-admin-allocation.md) (unassigned-пул). Контракт — [05-api-contracts.md](../05-api-contracts.md) §7 (`GET /admin/teams`), §5 (`/api/admin/teams*`). Референс — `mail-agregator/backend/app/templates/admin/groups/list.html`.

## Context

Страница `GET /admin/teams` в sms-агрегаторе сейчас — **пустая SSR-оболочка**: backend отдаёт только `csrf_token`, а список команд подгружается client-side (`admin_teams.js` → `fetch GET /api/admin/teams`, затем ручной рендер `<table>`). Это расходится с проектом-референсом mail-agregator, где аналогичная страница команд (`templates/admin/groups/list.html`) — **полностью server-rendered**: единая таблица, где **каждая команда = отдельный `<tbody>`** из двух строк:

1. **мета-строка** команды: название, лидер, число участников, дата создания, действия;
2. **строка-список** (`<td colspan=...>`), разворачивающаяся на всю ширину и перечисляющая **ресурсы команды** (в mail-agregator — email-аккаунты; в sms — **телефонные номера**).

Требование заказчика: привести `/admin/teams` к тому же виду, что groups-страница mail-agregator, и **показывать под каждой командой её номера** (по текущей принадлежности `phone_numbers.team_id`), не потеряв существующие контролы (создать / переименовать / удалить / назначить лидера).

Текущая полностью клиентская страница также деградирует без JS (пустая таблица), тогда как SSR даёт работающий просмотр без JS — консистентно с `/messages` (ADR-0014) и `/admin` (ADR-0015).

## Decision

**`GET /admin/teams` переводится на SSR по образцу `mail-agregator/.../admin/groups/list.html`** — обязательное приёмочное условие (acceptance criterion) для frontend и reviewer.

### 1. Единая таблица, `<tbody>`-на-команду

Одна `<table class="admin-teams-table">` (порт `admin-groups-table` референса) в обёртке `<div class="table-wrapper">` с единым `<thead>`. Колонки строго в порядке: **Название · Лидер · Участников · Номеров · Создана · Действия** (6). Колонка «Номеров» — sms-специфичное дополнение (в референсе групп её нет: там число аккаунтов выводится в строке-списке); оставляем её, т.к. `GET /api/admin/teams` уже отдаёт `numbers_count` и текущая клиентская таблица её показывает — паритет по **структуре** (мета-строка + строка-ресурсов на `<tbody>`), а состав колонок мета-строки — надмножество референса на одну sms-колонку.

Каждая команда — **`<tbody>`** из двух `<tr>`:
- **мета-строка** (`admin-teams-table__row`, `data-team-id`): Название · Лидер · Участников · Номеров · Создана · Действия;
- **строка-номеров** (`admin-teams-table__numbers-row`) с единственной `<td colspan="6">`, содержащей список номеров команды (эффективный лейбл + номер + статус активности), либо empty-state «Номеров в команде пока нет.».

Браузер парсит каждый `<tbody>` как отдельную секцию — border-radius/spacing на команду без лишних обёрток (тот же приём, что в референсе).

### 2. Номера под командой — по ТЕКУЩЕЙ принадлежности

Список номеров каждой команды строится по `phone_numbers.team_id = <team.id>` (текущая принадлежность, не снимок — консистентно с [ADR-0014](./ADR-0014-sms-viewing-by-number-current-ownership-cursor-pagination.md)). Для каждого номера показывается **эффективный лейбл** `label or phone_number` (по образцу mail-agregator `effective_account_label = display_name or email`, ADR-0020; в sms — Feature «никнейм номера», [05](../05-api-contracts.md) §6), сам `phone_number` и признак `is_active` (неактивный — приглушённым стилем). Номера — read-only на этой странице (управление принадлежностью — `/admin` секция номеров + `PATCH /api/admin/numbers/{id}`; редактирование лейбла — `PATCH /api/numbers/{id}`).

### 3. Нераспределённые номера — вне scope teams-страницы

Номера с `team_id IS NULL` (unassigned-пул, [ADR-0009](./ADR-0009-unassigned-numbers-admin-allocation.md)) **не показываются** на `/admin/teams` — это страница команд, а нераспределённые номера по определению не принадлежат ни одной команде и управляются в секции «Нераспределённые номера» на `/admin` ([ADR-0015](./ADR-0015-admin-users-visual-parity-with-mail-agregator.md) §Consequences, [05](../05-api-contracts.md) §7). Отдельной секции unassigned на teams-странице не вводится.

### 4. Контролы сохраняются; SSR + прогрессивное JS-обогащение

Все существующие контролы `admin_teams.js` **сохраняются** (не теряем функциональность), но перепроектируются под SSR-DOM:

- **Создать команду** — тулбар-кнопка → диалог → `POST /api/admin/teams {name}` ([05](../05-api-contracts.md) §5).
- **Переименовать** — действие мета-строки → диалог → `PATCH /api/admin/teams/{id} {name}`.
- **Удалить** — действие мета-строки → `DELETE /api/admin/teams/{id}` (только пустую по home; иначе `409 team_has_members`).
- **Назначить лидера** — действие мета-строки → диалог → `PATCH /api/admin/teams/{id}/leader {new_leader_user_id}`.

**no-JS fallback** (обязателен для create/rename/delete — как в референсе групп): инлайн-формы `POST` с `_method=PATCH`/`_method=DELETE` (MethodOverride уже whitelist-ит `^/api/admin/teams/\d+$`, `^/api/admin/teams/\d+/leader$`, `/api/admin/teams`). **Назначение лидера** остаётся **JS-обогащённым** диалогом: список кандидатов (участники команды) подгружается по требованию из `GET /api/admin/users` (как сейчас) — полноценный no-JS-путь потребовал бы инжектить состав каждой команды в SSR-контекст (bloat); лидер задаётся тем же `PATCH .../leader` и через API доступен без UI. Это осознанное сужение (не блокер): просмотр и базовые мутации работают без JS, редкая операция смены лидера — с JS.

`admin_teams.js` перестаёт **фетчить и рендерить список** (теперь SSR); сводится к прогрессивному обогащению диалогов и перехвату inline-форм мутаций. `GET /api/admin/teams` (JSON, §5) сохраняется для программного доступа и подгрузки кандидатов в лидеры.

### 5. Копируется визуальный паттерн, не семантика

Как и в [ADR-0015](./ADR-0015-admin-users-visual-parity-with-mail-agregator.md): портируется **структура/вёрстка** референса (единая таблица, `<tbody>`-на-команду, строка-ресурсов на `colspan`), но **семантика** — по sms-модели. В частности: диалог «Создать команду» содержит **только** поле `name` (команда создаётся orphan, лидер — по правилу «первый=лидер»; полей `leader_user_id`/`member_ids[]` референса **нет**, как уже зафиксировано в [05](../05-api-contracts.md) §7 для тулбар-кнопки на `/admin`).

### 6. Split-ownership SSR-контракт

Шаблон `app/api/templates/admin/teams.html` авторит **frontend**; SSR-контекст передаёт **backend** (`app/api/routers/admin_ui.py::admin_teams_page`). Литеральные имена всех контекстных переменных, их типы и структура зафиксированы в [05-api-contracts.md](../05-api-contracts.md) §7 (`GET /admin/teams`) — **ДО** параллельной реализации. Схема БД не меняется (презентация из существующих `teams`/`users`/`user_teams`/`phone_numbers`).

## Rationale

- Заказчик требует привести teams-страницу к виду groups-страницы референса; текущая клиентская страница — отклонение от эталона.
- SSR + `<tbody>`-на-команду — каноничная модель референса; она же даёт рабочий просмотр без JS (паритет с `/admin`, `/messages`).
- Номера под командой по текущей принадлежности консистентны с ADR-0014 (видимость SMS) и дают админу целостную картину «команда → её номера» на одной странице.
- Сохранение всех контролов + no-JS fallback для частых операций — без потери функциональности при переходе с клиентского рендера.

## Consequences

- **Контракт [05](../05-api-contracts.md) §7 `GET /admin/teams` перепроектирован**: пустой контекст (`{csrf_token}`) → нормативный SSR-контекст (`teams`, `numbers_by_team`, `csrf_token`, `is_super_admin`) с литеральными именами (split-ownership).
- **Backend** (`admin_ui.py::admin_teams_page`): собирает `teams` (id, name, leader-бриф, `members_count`, `numbers_count`, created_at, is_active) + `numbers_by_team` (группировка `phone_numbers` по `team_id`, эффективный лейбл). Переиспользует `TeamRepository.member_counts`/`numbers_counts` и `PhoneNumberRepository` (выборка assigned-номеров с группировкой в Python).
- **Frontend**: `app/api/templates/admin/teams.html` переписывается на SSR-таблицу (`admin-teams-table`, `<tbody>`-на-команду, строка-номеров на `colspan`); `admin_teams.js` теряет ветку fetch+render списка, сохраняет диалоги/мутации + inline no-JS формы; CSS-блоки портируются из `admin-groups-table` референса.
- **Схема БД не меняется.**
- **QA**: приёмочный чек-лист — §7 «Паритет `/admin/teams` (нормативно)».

## Alternatives

- **Оставить клиентский рендер списка** — отклонено: расходится с референсом и деградирует без JS.
- **Показывать нераспределённые номера отдельной секцией на teams-странице** — отклонено (см. §3): их место — секция номеров на `/admin`; teams-страница — только про команды.
- **Полный no-JS путь для назначения лидера** (инжект состава команд в контекст) — отклонено как несоразмерное (bloat SSR-контекста ради редкой операции); лидер задаётся через API/JS-диалог (§4).
