## Context

Требуется распределять пользователей по **командам** и разграничивать доступ. Референс mail-agregator реализует роли-константы `super_admin`/`group_leader`/`group_member` и таблицу групп с лидером. Наш домен — «команды», а не «группы»/«проекты». Нужно:

- разграничить: кто управляет пользователями/командами (админ) и кто работает в команде;
- определить, кто становится лидером команды;
- максимально переиспользовать auth/admin-код референса;
- не усложнять модель раньше времени (нет требования multi-team).

Текущая SQLite-модель: `projects` + `user_project_access` (M:N доступ, без ролей). Auth = Basic/token без пользователей.

## Decision

1. **Роли-константы сохраняются из mail-agregator** ради переноса auth-кода 1:1: `users.role IN ('super_admin','group_leader','group_member')`. В UI — ярлыки **«Админ / Лидер / Участник»**.
2. **Доменный словарь** переименован под сервис: таблица `teams` (было `projects`), колонки `teams.leader_user_id`, `users.team_id`.
3. **Инвариант роль↔команда** (CHECK в БД, [04-data-model.md](../04-data-model.md)):
   - `super_admin` ⇒ `team_id IS NULL` (ровно один, из `.env`, seed);
   - `group_leader`/`group_member` ⇒ `team_id IS NOT NULL`.
4. ~~**Multi-team не вводится:** один пользователь = одна команда (`users.team_id`), без M:N. (`TD-003`.)~~ **Отменено [ADR-0012](./ADR-0012-multi-team-membership.md) (2026-07-03):** введена аддитивная M:N `user_teams`; `users.team_id` остаётся **домашней** командой (источник инварианта лидера и «первый=лидер»), `user_teams` — источник истины адресации SMS и видимости номеров. `TD-003` закрыт.
5. **Правило «первый=лидер»:** команда создаётся «пустой» (`leader_user_id IS NULL`); при добавлении **первого** участника он получает `role='group_leader'` и записывается в `teams.leader_user_id` (`teams_service.set_leader_if_absent`). Последующие участники — `group_member`.
6. **Циклический FK** `teams.leader_user_id ↔ users.team_id` разрешается через `users.team_id DEFERRABLE INITIALLY DEFERRED` (проверка до COMMIT), плюс constraint-триггер лидерства как defense-in-depth.
7. **Жизненный цикл лидерства и расформирование команды** (устраняет дедлок «лидер — участник, поэтому команда никогда не пуста»):
   - **Переназначение лидера** — `PATCH /api/admin/teams/{id}/leader` → `teams_service.set_leader(team_id, new_leader_user_id)` (кандидат обязан быть участником команды): в одной транзакции старый лидер → `group_member`, новый → `group_leader`, `teams.leader_user_id` обновляется. Это единственный способ сменить лидерство без роспуска команды.
   - **Перенос/удаление лидера, пока в команде есть другие участники** — запрещено (`409 leader_move_forbidden` для `PATCH /api/admin/users/{id}`, `409 user_is_leader` для `DELETE /api/admin/users/{id}`); сначала переназначить лидера.
   - **Лидер как единственный участник** — при переносе (`PATCH user`) или удалении (`DELETE user`) транзакция обнуляет `teams.leader_user_id` (команда становится пустой/orphan), не нарушая CHECK.
   - **Удаление команды** — `DELETE /api/admin/teams/{id}` только для полностью пустой команды (`leader_user_id IS NULL` и нет `users.team_id=id`); иначе `409 team_has_members`. `ON DELETE SET NULL` на `users.team_id` — safety-net, при штатном flow не срабатывает. Порядок disband — [05-api-contracts.md](../05-api-contracts.md) §5.
8. **Единственный super_admin защищён в БД** — partial-UNIQUE `users_single_super_admin` (`WHERE role='super_admin'`); `seed_admin` при смене `ADMIN_LOGIN` переименовывает существующего админа, а не создаёт второго (см. [04-data-model.md](../04-data-model.md), [08-security.md](../08-security.md) §1).
9. **Права (MVP):**
   - `super_admin` — CRUD пользователей/команд, reset/delete, видит всё;
   - `group_leader` — полноценный участник; отдельных привилегий над людьми на MVP нет (управление людьми — у админа);
   - `group_member`/`group_leader` — добавляют/удаляют номера своей команды ([ADR-0005](./ADR-0005-sms-addressing-via-team.md)), получают SMS команды.

## Rationale

- **Переносимость:** идентичные строки-константы ролей позволяют перенести `auth_service`/`admin_service`/`deps`/guards из mail-agregator почти без правок.
- **Понятность домена:** имена таблиц/колонок (`teams`, `team_id`, `leader_user_id`) отражают предметную область; UI-ярлыки на русском.
- **Простота:** один пользователь = одна команда снимает необходимость в M:N и per-group ролях; инварианты выражаются табличным CHECK.
- **«Первый=лидер»** — простое детерминированное правило без отдельного шага «назначить лидера»; у команды всегда либо нет участников, либо есть ровно один лидер.

## Consequences

**Плюсы:** минимум кода переноса, строгие инварианты в БД, предсказуемое лидерство.

**Минусы / издержки:**
- Расхождение «константа роли ↔ доменное имя» (`group_*` vs «команда») — нужно помнить маппинг; зафиксировано здесь и в [04](../04-data-model.md)/[05](../05-api-contracts.md).
- Управление лидерством требует явных операций: переназначение (`PATCH /api/admin/teams/{id}/leader`) при наличии других участников, обнуление лидера при переносе/удалении единственного участника, удаление только пустой команды. Отражено в контрактах (`409 leader_move_forbidden`, `409 user_is_leader`, `409 team_has_members`) и §7 выше. FK `teams.leader_user_id ON DELETE RESTRICT` требует, чтобы транзакция сперва обнуляла `leader_user_id`, затем удаляла пользователя.
- `super_admin` не входит в команду ⇒ не получает SMS (`Q-TG-1`).
- Смена «домашней» команды (`PATCH /api/admin/users/{id}` с `team_id`) должна соблюдать инвариант и правило «первый=лидер» в целевой команде.

## Alternatives

1. **Ввести собственные имена ролей (`admin`/`leader`/`member`).** Отклонено: сломало бы прямой перенос auth-кода mail-agregator; выгода косметическая (UI-ярлыки и так свои).
2. ~~**Multi-team (M:N `user_teams`).** Отклонено на MVP: нет требования; усложняет инварианты, видимость и адресацию. Зарезервировано как `TD-003`.~~ **Пересмотрено [ADR-0012](./ADR-0012-multi-team-membership.md):** требование подтверждено продуктом; введена аддитивная M:N `user_teams` (home = `users.team_id` + доп. членства), инвариант лидера остаётся на домашней команде.
3. **Явное назначение лидера админом.** Отклонено: лишний шаг и состояние «команда с участниками без лидера»; «первый=лидер» проще и всегда консистентно.
4. **Пер-командные роли.** Отклонено: роль глобальна (одна на пользователя); пер-командные роли не нужны при single-team.
