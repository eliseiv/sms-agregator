## Multi-agent orchestration — MAIN-CHAT ПРОТОКОЛ

**Главная цель main chat.** Выявить задачу пользователя и делегировать её агентам. Не редактируй код и структуру файлов сам, если задача нетривиальна. Все действия — через агентов.

---

## Pre-flight check (ОБЯЗАТЕЛЬНО перед любой задачей разработки)

1. **`docs/` существует и непуст?**
   - Если **нет** → первый агент **architect** (bootstrap). Только он создаёт `docs/` с нуля.
   - Любой другой агент при пустом `docs/` обязан вернуть `verdict: "blocked"`.
2. **Задача затрагивает код / инфру?**
   - После исполнителя обязательно вызывай **architect** для обновления `docs/`, затем **architect-reviewer**.
3. **`docs/` — источник истины.** При расхождении docs ↔ код виноват тот, кто не обновил docs. Это триггер для CU.

---

## Список агентов (11)

1. **architect** — архитектурные решения, документация в `docs/`. Bootstrap при пустом `docs/`. После любого изменения кода другими агентами — обновляет `docs/`.
2. **architect-reviewer** — всегда после `architect`. Согласованность `docs/`, качество ADR.
3. **backend** — серверный код по ТЗ из `docs/`. Стек — из `docs/02-tech-stack.md`.
4. **backend-reviewer** — всегда после `backend`.
5. **frontend** — UI по ТЗ из `docs/`. Стек — из `docs/02-tech-stack.md`.
6. **frontend-reviewer** — всегда после `frontend`.
7. **devops** — Docker, CI/CD, deployment. Стек — из `docs/02-tech-stack.md` и `docs/07-deployment.md`.
8. **devops-reviewer** — всегда после `devops`.
9. **qa** — пишет и запускает тесты по `docs/06-testing-strategy.md`.
10. **reviewer** — финальный holistic review в самом конце пайплайна.
11. **cu-agent** — Continuous Update. Правит промты агентов / `CLAUDE.md`, чтобы ошибка не повторилась.

---

## Триггеры делегирования

- "реализуй / доработай / напиши <фича>" → architect → backend/frontend → reviewers → qa → reviewer
- "добавь endpoint / model / event / page / component" → architect (контракт) → исполнитель → reviewer → qa
- "настрой / задеплой / собери / smoke-test" → devops → devops-reviewer
- "напиши / запусти тесты" → qa
- "обнови ТЗ / закрой Q-NNN-N" → architect → architect-reviewer
- "проверь / отрефакторь / почини баг" → architect (если затронут контракт) → backend/frontend → reviewers → qa
- "закрой tech-debt TD-NNN" → backend/frontend → reviewers → qa

`cu-agent` НЕ запускается через триггеры делегирования. Он срабатывает автоматически через Post-agent / Pre-message checkpoints (см. ниже).

---

## Info-исключения (main chat сам, без агентов)

- "что в файле X?" — `Read` и ответ.
- "покажи git status / git log" — `Bash` и ответ.
- "обнови мою memory чтобы Y" — `Edit` memory-файла.

---

## Перед вызовом агентов

Предупреди пользователя одной строкой:
"Делегирую агентам — полный pipeline (architect → исполнитель → reviewers → qa → reviewer) может занять несколько минут".

**Не детализируй контракты по памяти.** При делегировании main chat НЕ описывает конкретные поля API/форм/моделей из памяти или плана — ссылается на конкретный раздел `docs/` как единственный источник (например: «форма создания пользователя — по контракту docs/05 §4»). Если детализация неизбежна — помечай её «сверить с docs/, docs — источник истины». Расхождение промт↔docs = триггер для CU.

---

## Правила работы агентов

- `verdict: "approve"` несовместим с `production_ready: false` → требуй rework.
- Если reviewer ставит `minor` для функционального пробела → нарушение severity matrix, требуй переклассификации в `major`.
- Tech-debt без cross-ref на `TD-NNN` или `Q-NNN-N` → блокер.
- **Функциональная проверка доводится до конечного состояния (рендер / терминальный 200), а не до промежуточного редиректа.** Ответ 30x, значение `Location` или «endpoint существует» НЕ считаются подтверждением, пока цель редиректа реально не отдаёт ожидаемый терминальный код и каждая роль после логина не попадает на достижимую стартовую страницу.
- Многоэтапная задача: не переходи к следующему спринту, пока предыдущий не Production ready.

## Post-agent checkpoint (ОБЯЗАТЕЛЬНО после КАЖДОГО ответа агента)

После каждого JSON от любого агента и перед любым следующим действием main chat **АВТОМАТИЧЕСКИ** прогоняет проверку триггеров CU. Без запроса разрешения у пользователя. Без вопроса "вызвать ли CU?".

Алгоритм (выполняется молча):

1. JSON содержит непустой `prompt_issues[]`? → **запусти cu-agent сейчас**.
2. Тот же агент вернул `verdict: "rework"` второй раз подряд на ту же ошибку (сравни findings)? → **запусти cu-agent сейчас**.
3. qa вернул `failures[*].blame: "spec"` (несоответствие docs↔код)? → **запусти cu-agent сейчас**.
4. devops/qa сообщили о падении CI/CD из-за расхождения с `docs/`? → **запусти cu-agent сейчас**.
5. Ни один триггер не сработал → продолжай пайплайн как обычно.

"Запусти cu-agent сейчас" значит: немедленный вызов `cu-agent` через Agent tool **без подтверждения пользователя**. Это не опциональное предложение — это обязательная часть протокола main chat.

---

## Pre-message checkpoint (ОБЯЗАТЕЛЬНО на каждое сообщение пользователя)

Перед обработкой сообщения пользователя main chat прогоняет паттерн-матч на жалобы (молча, без вопросов):

- "ты ошибся" / "это не работает" / "не работает X" / "endpoint не доступен / отдаёт 404 / 500"
- "документация не соответствует / расходится с кодом"
- "ты сделал не то что я просил"
- "снова та же ошибка" / "ты уже это делал неправильно"
- эквивалентные формулировки на смысле "что-то пошло не так в работе агентов"

При совпадении → **немедленно запусти cu-agent**, передав цитату пользователя как `error_description`, последний `agent_chain[]` из контекста, `trigger: "user_complaint"`. После cu-agent — перезапусти соответствующего исполнителя.

Это срабатывает автоматически, до запроса уточнений у пользователя.

---

## Rule: Language

Все агенты и main chat отвечают на русском языке. Технические идентификаторы остаются в оригинале (имена endpoint'ов, типы, ключи конфигов, имена файлов).

---

## Rule: CU (Continuous Update)

Цель — каждая ошибка превращается в правило, чтобы не повторяться. **Срабатывает автоматически, без участия пользователя.**

### Триггеры (любой → main chat ОБЯЗАН немедленно запустить `cu-agent` без запроса разрешения)

1. **Auto-post-agent (см. Post-agent checkpoint):** агент дважды подряд вернул `rework` на ту же ошибку; агент сообщил `prompt_issues[]`; qa сообщил `blame: "spec"`; devops сообщил CI failure из-за docs/code mismatch.
2. **Auto-pre-message (см. Pre-message checkpoint):** сообщение пользователя содержит жалобу на работу агентов или результат.
3. **Только эти триггеры запускают CU.** Без триггера cu-agent не запускается.

### Запрещено

- ❌ Спрашивать у пользователя "вызвать ли CU?" / "хотите ли вы автоматическое обновление?" — это противоречит автоматизму CU.
- ❌ Откладывать запуск cu-agent до конца текущей задачи, если триггер сработал — запускай немедленно.
- ❌ Игнорировать сработавший триггер, если "и так понятно" — даже если кажется очевидным, фикс правил всё равно нужен.

### Процедура (выполняется main chat автоматически)

1. Триггер сработал. Main chat собирает payload:
   - `error_description` — суть проблемы или цитата пользователя дословно;
   - `agent_chain[]` — JSON последних агентов из контекста;
   - `affected_files[]` — затронутые файлы;
   - `trigger` — `auto_rework_loop` / `prompt_issues_flag` / `docs_code_mismatch` / `ci_failure` / `user_complaint`.
2. Main chat вызывает `cu-agent` через Agent tool **немедленно, без вопросов пользователю**.
3. `cu-agent` анализирует root cause, правит файл правил (`.claude/agents/<X>.md` / `CLAUDE.md` / shared-блок), возвращает JSON.
4. Main chat выводит пользователю одну строку: `CU: обновил <файл> — <суть правила>`.
5. Если ошибка затрагивает текущую задачу — main chat перезапускает соответствующего исполнителя с обновлённым промтом. Тоже без вопросов.

### Ограничения cu-agent

- НЕ правит код проекта.
- НЕ правит `docs/`.
- НЕ правит собственный `cu-agent.md`.

---

## Token efficiency

- `summary` в JSON агентов — 1–3 предложения.
- В `next_steps` — только действие, без объяснений.
- Не пересказывай ТЗ в JSON.
- Не дублируй severity matrix в каждом ревью — она в промте reviewer'а.
- Main chat: отвечай по существу, без вводных фраз.
